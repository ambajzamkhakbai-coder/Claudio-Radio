## 2. 项目概述 & MVP 范围

### 2.1 产品定位
一款运行在本地服务器、通过 PWA 界面交互的**个人 AI 音乐电台**。它能够像真正的电台 DJ 一样，结合用户品味、当前时间、天气及日程，通过 Gemini 动态组装 Prompt 并生成语音主持旁白，无缝串联网易云音乐的音频流播放。

### 2.2 核心模块（MVP 范围）
* **第一层：外部上下文 (Context & Tools)**：用户语料初始化、Gemini API 接入、网易云音乐 API、第三方服务（天气、飞书日程）。
* **第二层：本地大脑核心 (Core Logic)**：意图分流、提示词动态组装、大脑适配器、节律调度器、语音管道（TTS）、状态持久化。
* **第三层：运行时上下文窗口 (Context Window & Model)**：6 片 Prompt 碎片聚合、Gemini 推理、语音合成与播放队列管理。
* **第四层：交互表层 (UI & HTTP Contract)**：PWA 播放器前端、本地 HTTP/WebSocket 协议通信。

## 3. 全局业务流程与状态机

### 3.1 核心业务流程图
```
[节律调度/用户触发] 
       │
       ▼
[收集上下文] ──► (系统提示词 / 用户语料 / 环境天气日程 / 历史记忆 / 工具结果)
       │
       ▼
[Context 组装] ──► 生成完整 Prompt 投喂给 Gemini
       │
       ▼
[Gemini 推理] ──► 输出 JSON: { say: "旁白文本", play: ["歌曲ID"], reason: "推荐理由" }
       │
       ▼
[异步管道分流]
 ┌─────┴────────────────────────┐
 ▼                              ▼
[Fish Audio TTS 异步合成]    [网易云 API 获取音频流]
 └─────┬────────────────────────┘
       │ (通过本地缓存/预加载队列同步)
       ▼
[PWA 播放器] ──► 播放 DJ 旁白 -> 播放音乐 -> 状态上报
```

### 3.2 播放器音频状态流转机
* **Idle（空闲）**：等待节律调度（如早间电台触发）或用户手动点击。
* **Loading（加载中）**：Gemini 正在推理，或 TTS 正在合成，前端执行预加载（prefetch 10s）。
* **Speaking（DJ 播报中）**：播放 Fish Audio 合成的语音旁白，音乐背景音自动淡入（Ducking 效果，音量降至 20%）。
* **MusicPlaying（音乐播放中）**：旁白结束，音乐音量恢复 100%。
* **Paused（暂停）**：用户手动暂停音频。
* **Exception（异常兜底）**：API 请求超时或网易云无版权，自动跳过并记录。

## 4. 功能需求详述

### 4.1 第一层：外部上下文 (External Context)

#### 4.1.1 用户产品味语料库初始化
* **功能说明**：本地服务器读取指定 Markdown/JSON 文件，作为 AI 认识用户的基础。
* **物理文件定义**：
    * `taste.md`：用户音乐偏好描述（喜欢摇滚、华语流行、特定歌手等）。
    * `routines.md`：日常作息时间表。
    * `playlists.json`：核心歌单资产。
    * `msc-rules.md`：电台运行守则（如“每 3 首歌必须插播一次天气”）。
* **测试点**：修改本地 md 文件后，验证系统重启或热加载时，模型读取的内容是否实时同步更新。

#### 4.1.2 基础设施 API 接入
* **Gemini 接入**：作为核心大脑，配置本地 API Key，使用流式或标准 JSON Output 模式。
* **NeteaseCloudMusicApi 接入**：
    * `/search`：歌曲/歌单搜索。
    * `/song_url`：获取播放音频流。
    * `/lyric`：歌词获取。
    * `/recommend`：每日推荐/个性化推荐雷达。
* **Voice / 1/O 接入**：
    * **Fish Audio (TTS)**：输入文本，输出 `.mp3` 音频流。
    * **OpenWeather**：获取当前城市天气（晴/雨/温度）。
    * **飞书日程 (Lark)**：读取当天 Calendar 事件。

---

### 4.2 第二层：本地大脑核心 (Local Brain)

#### 4.2.1 意图分流 (`router.js`)
* **逻辑定义**：解析用户输入或调度指令。
    * **简单指令直连**：如用户在 PWA 界面点击“下一首”或输入“放首周杰伦”，绕过 Gemini 大脑推理，直连网易云 API 响应。
    * **自然语言意图**：输入“心情不好，放点治愈的”，分流至 Gemini 大脑进行上下文组装。

#### 4.2.2 提示词组装与大脑适配 (`context.js` / `claudio.js`)
* **逻辑定义**：
    * `context.js`：将第一层的文件内容与第三层的实时环境拼接。
    * `claudio.js`：使用 `spawn` 子进程或标准库调用 Gemini。强制要求 Gemini 严格输出固定格式的 JSON（严禁夹带自然语言包裹包裹）。
* **边界异常**：若 Gemini 返回非标准 JSON，系统需触发本地正则清洗，若清洗失败则进入兜底策略（直接播放网易云推荐歌单，不播报旁白）。

#### 4.2.3 节律调度器 (`scheduler.js`)
* **逻辑定义**：基于 Cron 表达式或本地时钟挂钩（Hook）。
    * **示例场景**：`07:00` 触发早安电台（天气+今日日程+轻快音乐）；`23:00` 触发小酒馆模式（白噪音+舒缓音乐）。

#### 4.2.4 声音管线 (`tts.js`)
* **逻辑定义**：
    * 接收 `claudio.js` 生成的 `say` 文本。
    * 调用 Fish Audio API 转换为 `.mp3`。
    * **缓存机制**：本地建立缓存目录 `/cache/tts/* .mp3`。文件名采用文本的 MD5 值的哈希 `/{hash}.mp3`。若相同文本再次触发，直接读取本地缓存，减少 API 消耗。

#### 4.2.5 状态与记忆持久化 (`state.db`)
* **数据定义**：使用 SQLite 或本地 JSON 文件存储：
    * `messages`：与 AI 的对话历史。
    * `plays`：历史播放记录（已听歌曲 ID、时间、跳过率）。
    * `plan & prefs`：当前运行计划与用户临时偏好变更。
* **测试点**：断电重启后，电台能从 `state.db` 恢复断点，继续播放上一首未完的歌单。

### 4.3 第三层：运行时上下文窗口 (Runtime Context Window)

#### 4.3.1 组装盒子（6 片贴成 Prompt）
每次向 Gemini 发起推理请求时，必须动态打包以下 6 类信息片段：
1.  **系统提示词**：`prompts/dj-persona.md`（定义电台 DJ 的人设、语调、说话字数限制）。
2.  **用户语料**：`user/*.md`（用户的品味、习惯）。
3.  **环境注入**：`weather` + `calendar` + `now`（当前时间）。
4.  **已检索记忆**：`state.db` 中的 `plays` 历史（防止 1 小时内推荐重复歌曲）。
5.  **用户输入/工具结果**：`/api/chat` 的对话内容或 `ncm search` 结果。
6.  **执行轨迹**：`scheduler` 触发源信息或 `webhook` 回调。

#### 4.3.2 模型输入输出契约 (Model Interface Contract)
* **Model Method**: `compute(fragments)`
* **Output JSON Schema**:
```json
{
  "say": "大家早上好，今天是公历5月22日，旧金山外面正下着小雨。看到你9点有一场产品评审会，顶住压力，先来一首苏打绿的《Rain》润润耳朵。",
  "play": [
    {"id": "5264641", "source": "ncm"}
  ],
  "reason": "根据下雨天气与后续高压会议，推荐舒缓且切题的曲目",
  "segue": "fade-out"
}
```

### 4.4 第四层：交互表层 (Interactive Surface)

#### 4.4.1 PWA 前端界面需求 (`localhost:8080`)
* **视觉与交互规范**：
    * **结构**：采用独立播放器视窗设计。
    * **核心视图**：Player（主播放组件）、Profile（用户画像/语料查看）、Settings（API Key 密钥配置）。
    * **音频架构**：前端使用单 `<audio>` 标签，通过控制 `src` 切盘来实现旁白与歌曲的无缝衔接。
* **性能要求 (Prefetch)**：
    * 当前歌曲播放剩余 **10秒** 时，前端通过 WebSocket 向服务器发送预载指令（prefetch）。
    * 后端提前请求下一首歌曲的 `song_url` 以及生成下一段 TTS 音频，确保切歌时“零卡顿”。
* **离线与缓存**：配置 Service Worker (sw)，缓存前端静态资源及已播放音频片段。

#### 4.4.2 HTTP/WebSocket 通信契约 (PWA ↔ Server)
后端需严格支持以下 6 条核心核心通信线：

```
┌─────────────────────────────────────────────────────────────┐
│                       HTTP REST API                         │
├──────────────────┬──────────────────────────────────────────┤
│ POST /api/chat   │ 用户在 PWA 输入框与 AI 的实时对话指令    │
├──────────────────┼──────────────────────────────────────────┤
│ GET /api/now     │ 获取当前播放状态（歌曲、进度、当前旁白） │
├──────────────────┼──────────────────────────────────────────┤
│ GET /api/next    │ 强制切歌，触发下一轮大脑计算             │
├──────────────────┼──────────────────────────────────────────┤
│ GET /api/taste   │ 读取/更新本地用户品味语料库 (taste.md)   │
├──────────────────┼──────────────────────────────────────────┤
│ GET /api/plan/today│ 获取今日已生成的电台节目单（时序表）  │
└──────────────────┴──────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────────┐
│                      WebSocket (WS)                         │
├──────────────────┬──────────────────────────────────────────┤
│ WS /stream       │ 推送实时播放事件：TTS正在合成、当前播放中、│
│                  │ 歌曲即将结束(Trigger Prefetch)、音量控制 │
└──────────────────┴──────────────────────────────────────────┘
```

## 5. 边界异常与非功能性需求

### 5.1 边界情况处理 (Edge Cases)
* **网易云无版权/播放失败**：当 `/song_url` 返回空或 404 时，`router.js` 需立刻捕获，自动向上游重试当前 `play` 列表中的备用歌曲，或重新调用网易云推荐接口。
* **TTS 生成超时**：设定 TTS 请求超时限制为 5 秒。若超时未返回 `.mp3`，PWA 界面应跳过旁白，无缝进入音乐播放，确保音频流不中断。
* **网络断开**：PWA 端利用本地 Service Worker 兜底，播放本地缓存的白噪音或历史音频，并提示“电台信号流失中”。

### 5.2 性能与体验指标
* **推理延迟控制**：从触发 Prefetch 到后端组装 Prompt 并完成 Gemini 推理，耗时需控制在 2.5s 以内。
* **音频衔接效果**：旁白结束切换到歌曲时，音频应有 500ms 的淡入淡出（Crossfade）过渡，避免声音突变破坏电台沉浸感。
